# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & dev

Wails v2 desktop app — Go backend, vanilla JS frontend (no bundler). Requires Go 1.25+, Node 18+, and Wails CLI:

```sh
go install github.com/wailsapp/wails/v2/cmd/wails@latest
go mod tidy

wails dev                                    # hot-reload dev (uses frontend/build.js --watch)
wails build                                  # local-arch build
wails build -platform darwin/universal       # universal Mac
wails build -platform windows/amd64
wails build -platform linux/amd64            # needs WebKit2GTK
```

Output lands in `build/bin/`. The frontend has no test runner and no Go tests exist.

The frontend "build" (`frontend/build.js`) just copies `index.html` + `src/` into `dist/`. Wails embeds `frontend/dist` via `//go:embed all:frontend/dist` in [main.go](main.go), so the dist folder must exist (even if empty) for `go build` to succeed standalone.

As of v1.1.0 there are NO runtime tool dependencies. yt-dlp ships embedded; the video+audio stream merge that previously needed ffmpeg is done by the pure-Go muxer in [muxer.go](muxer.go). See the "Editor + capability gating" section.

## Architecture

### Wails binding model
The `App` struct in [app.go](app.go) is bound via `Bind: []interface{}{app}` in [main.go](main.go). Every public method on `*App` becomes an async function on `window.go.main.App.<Name>` in the frontend. DTOs returned from these methods (e.g. `ChannelInfo`, `VideoInfo`, `EditCapability`) become the JS object shapes — when adding fields, mind the JSON tags. Wails-generated TS/JS bindings live under `frontend/wailsjs/` (regenerated by `wails dev`/`wails build`).

The frontend also uses the Wails event bus: `runtime.EventsEmit(ctx, "ytdlp-progress", ...)` from Go is consumed via `EventsOn("ytdlp-progress", cb)` in JS. yt-dlp stdout/stderr are streamed line-by-line through this channel.

### Custom asset-server handler
[video_handler.go](video_handler.go) installs a `MediaHandler` as the Wails `assetserver.Options.Handler`. It intercepts:

- `/video/<rel-path>` — `http.ServeFile` so HTTP Range works (required for `<video>` seeking)
- `/thumb/<rel-path>` — disk cache → memory fallback → 404

Anything else falls through to the embedded frontend bundle. `safeJoin` rejects paths that escape the library root (defense against `../` in URLs). When adding a new media URL prefix, route it here.

### Library model
[library.go](library.go) implements a filesystem layout of `Videos/<Channel>/[<Folder>/]<Video>.{mp4,m4v,mov}`. **Folders nest exactly one level deep** — `scanFolder` does not recurse into subdirectories of folders, and `validateChannelName` is reused for folder names so the same cross-platform character rules apply. `findVideosDir` locates `Videos/` next to the binary by walking up to 5 levels (handles `.app` bundles and Linux AppImage via `$APPIMAGE`). Hidden folders/files are skipped everywhere, so `.thumbs/`, `.trash/`, `.state.json`, `._*` AppleDouble forks, etc. never appear in the UI. MP4 metadata (duration + dimensions) is read directly from the `moov` box via `mp4ff` with `DecModeLazyMdat` — no codec decoding.

Channels with zero videos are still included in the scan so newly-created empty channels (and folders) are visible. The `Channel` struct holds `Videos` (root-level) and `Folders` separately, but they're fused into a single A-Z sorted `[]Item` by `App.Items()` — folders interleave with videos, no group header. `Item.Kind` ("folder" | "video") is the discriminator the JS uses to render and route clicks.

### Thumbnails
[thumbnails.go](thumbnails.go) + [app.go](app.go) — discovery priority:

1. Embedded MP4 cover art (scans the serialized `moov` bytes for JPEG/PNG signatures rather than navigating the box tree, since `mp4ff`'s box API varies between versions)
2. Sidecar image: `{stem}.{jpg,jpeg,png,webp}` next to the video
3. Manual fetch from `i.ytimg.com` (no API key; tries `maxres → sd → hq` and skips the ~800 B "image not available" placeholder)
4. Manual import from a local file

Cache key is `sha1(absVideoPath)`. Cached files live in `Videos/.thumbs/<sha1>.jpg`. If that directory isn't writable (read-only USB), we fall back to an in-memory map (`App.memThumbs`). Any rename (channel or video) must call `migrateThumbCache` because the cache is keyed on the absolute path.

### Editor + capability gating
[editor.go](editor.go) exposes channel/folder/video CRUD plus `AddVideos` (yt-dlp invocation). The "edit mode" toggle in the UI is gated on `EditCapability.Enabled`, which requires yt-dlp + internet. Capability is cached for 8s and re-checked when the user clicks the badge (`RefreshEditCapability`).

**yt-dlp ships embedded.** [bundled/](bundled/) `go:embed`s a per-platform yt-dlp standalone binary (PyInstaller bundle, ~25 MB) and `App.resolveYtdlpPath` extracts it into `Videos/.bin/yt-dlp[.exe]` on first use, atomically (per-PID tmp + rename, size-equality skip-rewrite). On macOS the post-extract step strips `com.apple.quarantine` and applies an ad-hoc codesign so Gatekeeper doesn't kill the spawned process. The repo only commits 1-byte placeholders; `tools/fetch-ytdlp.sh` populates the real binaries before `wails build`, both locally and in the GitHub Actions release workflow. `bundled.HasEmbeddedYtdlp()` returns true only when the embed is `>= 1 MB`, so developer builds without the fetch step transparently fall back to PATH-based discovery (`findCommand`). If extraction into `Videos/.bin/` fails (read-only USB, library not loaded), the resolver tries `os.UserCacheDir()/YTDisc/bin` next, then PATH.

**ffmpeg is no longer required.** [muxer.go](muxer.go) is a pure-Go MP4 stream-copy muxer that combines the separate video-only `Title.f137.mp4` and audio-only `Title.f140.m4a` files yt-dlp leaves on disk into a single `Title.mp4`. It uses `mp4ff`'s box-tree representation: parses both inputs lazily (no full-file load), rewrites each track's `stco`/`co64` chunk-offset table to point at the new merged-file positions, then writes `[ftyp][mdat[video samples][audio samples]][moov]` with sample data stream-copied directly from the source files via `io.CopyN`. yt-dlp is invoked WITHOUT `--ffmpeg-location` so its own merger fails silently, leaving the orphan pair for `mergeOrphanStreams` to pick up — that path is now the only post-download merge step.

Folder operations:
- `CreateFolder` / `RenameFolder` / `DeleteFolder` — straightforward; `RenameFolder` snapshots the old absolute paths before `os.Rename` so cache migration can run after.
- `EmptyFolder` moves every video out into the channel root (with sidecars + thumb cache) and then removes the now-empty folder. Falls back to `RemoveAll` if `os.Remove` trips on a leftover `._*` AppleDouble fork.
- `MoveVideo(relPath, destFolder)` is intentionally **same-channel only** — no cross-channel moves. The destination is either a folder name in the current channel or `""` to mean channel root.

`AddVideos(channelName, destFolder, urls, quality)` runs yt-dlp once per URL. **`formatSelector`** picks the `-f` argument from the quality label ("fhd"|"hd"|"sd") — every variant forces H.264 (`vcodec^=avc1`) because WKWebView on macOS won't reliably play AV1 (format 399) or VP9. **Do not strip the avc1 constraint** when adjusting these selectors. After every download, `mergeOrphanStreams` scans for unmerged `Title.f137.mp4` + `Title.f140.m4a` pairs and merges them with `ffmpeg -c copy` — yt-dlp's merger sometimes silently fails on macOS Homebrew installs even with `--ffmpeg-location` set.

Playlists: any URL matched by `looksLikePlaylistURL` (a YouTube URL with `list=` or `/playlist`) takes a separate code path — `fetchPlaylistTitle` calls yt-dlp with `--flat-playlist --print %(playlist_title)s`, the title is sanitized (`sanitizeFolderName`) and uniquified (`uniqueFolderName`), the folder is created under the channel root, and yt-dlp runs with `--yes-playlist` and a `%(playlist_index)03d -` filename prefix so the saved files sort in playlist order rather than A-Z by title. Playlist folders **always go to channel root**, even if the user is currently inside a folder when adding — folders don't nest.

Thumbnails are written as sidecars (`--write-thumbnail`) rather than embedded, because yt-dlp's `ThumbnailsConvertor` postprocessor doesn't reliably honor `--ffmpeg-location` on macOS Homebrew installs (yt-dlp issue #8680). Our scanner discovers sidecars natively, so the UX is identical.

Deletes go to `Videos/.trash/<timestamp>_<name>` (recoverable). The trash dir is hidden so the scanner skips it.

### Resume playback (positions store)
[positions.go](positions.go) implements bookmarking the playhead so closing a video and reopening it later jumps back to where you left off. Persisted to `Videos/.state.json` (hidden), keyed by **forward-slash relative path** under `Videos/` (not absolute path like the thumbnail cache — so positions survive moving the library between machines). `App.positions` is a `*positionStore`; bound methods are `GetPosition` / `SavePosition` / `ClearPosition`.

The product rule "don't resume in the first 45 s or last 20 s" lives in **the frontend** ([main.js](frontend/src/main.js) `shouldResumeAt`). The backend just stores whatever it's told. The frontend saves on a 5 s timer while playing, on `pause`, on player-close, and on `beforeunload`/`pagehide` so ⌘W and app-quit are covered. `ended` clears the bookmark. Crashes/force-quits lose at most 5 seconds of bookmark — that's the documented behavior.

When a video is renamed or moved, `migrateThumbCache` also calls `positions.rename(oldRel, newRel)` so the bookmark follows the file. `DeleteVideo` clears the bookmark.

### macOS PATH fix
[pathfix.go](pathfix.go) runs in `init()` to prepend Homebrew/MacPorts/user-bin paths to PATH. macOS GUI apps launched from Finder or Dock inherit a minimal launchd PATH (no `/usr/local/bin` or `/opt/homebrew/bin`), which would otherwise make `yt-dlp` and `ffmpeg` invisible to the app even when installed. Only existing directories are added, no duplicates — so it's a no-op when launched from a terminal.

### Concurrency
`App.mu` is a `sync.RWMutex` guarding `videosDir`, `library`, and `memThumbs`. Hold `RLock` for reads, `Lock` for mutations. `App.positions` and `App.editCap` have their own internal mutexes — don't take `App.mu` to access them. The thumbnail discovery walk and yt-dlp event streaming run on goroutines; `Rescan` is called after every mutation to refresh the in-memory library.

## Conventions

- Filename validation in `validateChannelName`/`validateFileName` rejects `/ \ : * ? " < > |` — the library is meant to live on a USB stick that may move between macOS/Linux/Windows, so cross-platform-illegal characters are blocked even on Unix.
- Hidden files (`._*` AppleDouble forks on exFAT/FAT32, `.DS_Store`, `Thumbs.db`) are skipped everywhere — the scanner, the orphan-stream merger, etc. New filesystem walks should match.
- Frontend is plain ES modules, no bundler, no framework. Don't add one without a strong reason — the embedded `dist/` size matters because the binary ships on a USB stick.
